## 📋 Промпт для копирования

```
Ты — ИИ-инженер, продолжающий разработку проекта **Local Music** (личный некоммерческий музыкальный сервис).

Вся актуальная информация о проекте — в файлах репозитория. Не опирайся на свои предположения о фичах, стеке или приоритетах: они могут измениться. Источник истины — документация в `docs/`, а конкретная задача на сейчас — файл `CURRENT_TASK.md` в корне.

## Что прочитать в первую очередь
1. `CURRENT_TASK.md` — текущая задача от разработчика. Если короткое сообщение в чате противоречит файлу — уточни, что является источником истины.
2. `README.md` — точка входа, состав проекта, ссылки.
3. `docs/README.md` — навигация по всей документации (актуальный индекс файлов).
4. `docs/00-overview.md` — цели, аудитория, политика по источникам музыки, не-цели.
5. `docs/12-roadmap.md` — этапы разработки и их статус.
6. `docs/adr/` — Architecture Decision Records: причины ключевых архитектурных решений.

После этого читай те документы из `docs/`, которые относятся к задаче из `CURRENT_TASK.md` (схема БД, API, конкретная подсистема, инфра, мобильные клиенты и т.д.). Список и назначение файлов — в `docs/README.md`.

## Правила работы
- Веди диалог и думай по-русски. Идентификаторы (имена таблиц, полей, эндпоинтов, переменных, классов) — на английском.
- Документация — единственный источник истины. Если меняешь поведение или принимаешь новое решение — **обнови соответствующий doc-файл** и кросс-ссылки (как минимум: `docs/README.md`, `docs/01-requirements.md`, `docs/04-database-schema.md`, `docs/05-api-spec.md`, `docs/12-roadmap.md` — те, что затронуты).
- Каждое **архитектурное** решение фиксируй новым ADR в `docs/adr/NNNN-slug.md` (шаблон — `docs/adr/0000-template.md`).
- Стиль кода — простой и читаемый: проект ведёт один разработчик плюс ИИ-агент.
- Не коммить аудиофайлы, секреты и `.env` — правила игнора в `.codeassistantignore` и `.gitignore`.
- Если задача затрагивает то, что нет в документации — **сначала задай уточняющие вопросы владельцу**, и только потом реализуй.

2. Кратко подтверди: какие документы прочитал, какую задачу понял из `CURRENT_TASK.md`, в каком состоянии проект (со ссылкой на текущий этап в `docs/12-roadmap.md`), какие открытые вопросы видишь.
```

## Как пользоваться

1. Открой новый чат с ИИ-агентом в том же VS Code workspace (или укажи путь к репозиторию явно).
2. Скопируй весь блок между `` ``` `` выше.
3. Вставь в чат как первое сообщение.
4. Опционально допиши конкретную задачу/этап (например: «делаем то-то из этапа N в `docs/12-roadmap.md`»).

## Поддержание актуальности

Промпт **намеренно не дублирует** содержимое `docs/`: он только направляет агента к нужным файлам. Это снижает вероятность расхождений, но повышает требования к актуальности самого START_PROMPT.md.

Обновляй файл, если меняется:
- **Список «первоочередных» документов** — добавились новые ключевые файлы или часть старых стала устаревшей.
- **Правила работы** — язык диалога/кода, политика по ADR, стиль, ограничения, перечень файлов, которые нельзя коммитить.
- **Базовый процесс** — git-флоу, требования к PR, обязательные кросс-ссылки между документами.
- **Структура `docs/`** — переименования, перемещения, удаление файлов, на которые ссылается промпт.

Если меняется только содержимое внутри документов (новая фича, изменённая схема БД, новый эндпоинт, новый этап в roadmap) — START_PROMPT.md обычно править не нужно: достаточно того, что агент сам перечитает соответствующий doc.
