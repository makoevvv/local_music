# 11. Безопасность

> Проект некоммерческий и используется в узком кругу. Здесь — только технические аспекты безопасности.

## 1. Аутентификация
- Пароли хешируются через **argon2id** (рекомендация OWASP). Параметры: time=3, mem=64MiB, parallelism=1.
- JWT: `HS256` с секретом ≥ 32 байт; access — 60 мин, refresh — 30 дней.
- Refresh-токены — храним в БД в виде sha256-хеша (`refresh_tokens` таблица), поддерживается отзыв.
- Регистрация — только по invite-кодам (см. `INVITE_CODES` в `.env`).

## 2. Авторизация
- Роли: `admin`, `user`.
- Эндпоинты `/admin/*` — только admin.
- Доступ к плейлистам и трекам: владелец видит свои; публичные плейлисты доступны всем зарегистрированным; библиотека треков — общая для всех пользователей сервиса.

## 3. Транспорт
- HTTPS обязателен снаружи. Через **Cloudflare Tunnel** TLS обеспечивается автоматически.
- Внутри docker-сети — HTTP (изолированная bridge-сеть).
- HSTS включён в Caddyfile.

## 4. Защита API
- CORS: whitelist origins (web, iOS-симулятор для разработки).
- Rate-limit (см. [`05-api-spec.md`](05-api-spec.md:1) §11).
- Защита от brute-force на `/auth/login`: задержка + блок IP после N неудач (5 за 10 мин → 15 мин блок).
- Все входные данные — Pydantic-схемы.

## 5. Секреты
- Только в `.env`, никогда в git.
- В CI — через GitHub Secrets.
- Ротация JWT-секрета — раз в полгода (логаут всех).

### 5.1. Токены внешних сервисов (импорт)
- Access/refresh-токены провайдеров (`spotify`, `yandex_music`, `apple_music`, `youtube_music`) хранятся в `external_accounts` в виде **AES-GCM** ciphertext (`access_token_enc`, `refresh_token_enc`, `token_nonce`).
- Мастер-ключ — `EXTERNAL_TOKENS_KEY` (32 байта, base64) из `.env`; в код/логи **никогда** не попадает.
- Per-row nonce 12 байт генерируется `secrets.token_bytes(12)` для каждой записи/обновления.
- Ротация мастер-ключа поддерживается воркером `rotate_external_tokens` (re-encrypt all rows, держит старый ключ как `EXTERNAL_TOKENS_KEY_PREV` до завершения).
- В логах импорта пишем только `provider`, `external_user_id`, `status`; токены и `raw`-payload-ы внешних API маскируются.
- Подробности — [`17-external-imports.md`](17-external-imports.md:1).

## 6. Хранилище
- Файлы аудио вне корня репо.
- Доступ к `/tracks/{id}/stream` только аутентифицированным.
- Подписанные URL (на будущее, для CDN): JWT-токен с TTL 1ч на конкретный track_id.

## 7. Зависимости
- Регулярный `pip audit` / `npm audit` в CI.
- Образы Docker обновлять (база `python:3.12-slim`, `node:20-alpine`).

## 8. Логи и аудит
- В логах **не** пишем пароли, токены, тело запросов с чувствительными полями.
- Admin-действия → `audit_log` (см. [`04-database-schema.md`](04-database-schema.md:1)).

## 9. Резервное копирование и DR
- БД и storage бэкапятся ежедневно (см. [`09-infrastructure.md`](09-infrastructure.md:1)).
- Тест восстановления — раз в квартал.

## 10. Удаление аккаунта
- Soft-delete пользователя: `is_active=false`, плейлисты остаются, личный «Любимое» удаляется.
- Полное удаление — admin вручную, через CLI.

## 11. Чек-лист безопасности перед открытием доступа извне
- [ ] `.env` с уникальными секретами.
- [ ] Admin создан, default-аккаунтов нет.
- [ ] HTTPS работает, http→https редирект включён.
- [ ] Cloudflare Access (PIN/email) перед публичным URL (если не Tailscale).
- [ ] Открытые порты — только 443 наружу.
- [ ] Бэкапы запускаются по расписанию.
- [ ] Логи мониторятся (хотя бы `docker compose logs`).
