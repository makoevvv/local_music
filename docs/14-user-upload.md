# 14. Загрузка треков пользователем (manual upload)

Параллельно с поиском в интернете (см. [`06-music-sourcing.md`](06-music-sourcing.md:1)) пользователь должен иметь возможность загрузить собственный файл со своего устройства, указать метаданные и пройти проверку достоверности.

## 1. Сценарий

1. Пользователь открывает «Добавить трек» → вкладка **«Загрузить файл»** (вторая вкладка — «Найти в интернете»).
2. Drag-n-drop или выбор файла. Поддерживаемые форматы: `mp3, m4a, aac, flac, ogg, opus, wav` (≤ 50 МБ).
3. Backend читает теги (`mutagen`) → предзаполняет форму: **название, исполнитель, альбом, год, обложка** (если встроена).
4. Пользователь правит/дополняет:
   - **Название** — обязательно.
   - **Исполнитель(и)** — обязательно, поддержка нескольких + ролей (main/feat).
   - **Альбом** — необязательно. Если не указан — backend на этапе верификации **самостоятельно определит** принадлежность трека к альбому (см. §2). Статус сингла назначается только если трек действительно вышел как сингл.
   - **Жанры, язык, год** — опционально (можно из автодополнения).
   - **Обложка** — опционально, картинка ≤ 5 МБ; если не задана и в файле её нет — берём из MusicBrainz/Cover Art Archive, иначе плейсхолдер.
   - Флаг `instrumental` / `has_lyrics`.
5. Нажимает «Проверить». Backend параллельно запускает:
   - **верификацию достоверности** (см. §4),
   - **поиск трека в открытых источниках** (см. §3) — чтобы у записи в индексе был реальный публичный источник, а не только `user_upload`.
6. Показывается «карточка проверки» с уровнем доверия (`high/medium/low/none`), расхождениями и **списком найденных открытых источников** (YouTube/SoundCloud/Bandcamp). Пользователь может:
   - Принять предложенные метаданные.
   - Выбрать «опорный» источник из найденных (отметить, какой использовать как primary для восстановления).
   - Отправить как есть, если ничего не найдено (`force=true`).
7. Backend создаёт `Track(status='ready')`, сохраняет файл, обогащает метаданные, пишет записи в [`track_sources`](16-library-index-and-restore.md:1): и `user_upload`, и все подтверждённые публичные источники.

## 2. Автоопределение альбома (album resolution)

Если пользователь не указал альбом, backend **обязан попытаться сам найти** реальный релиз трека до того, как будет создан сингл-альбом. Это критично: статус сингла должен иметь только тот трек, который действительно выходил как сингл.

### Алгоритм

1. **AcoustID** уже отдал mbid recording'а (если уровень верификации ≥ medium). По mbid recording'а запрашиваем у MusicBrainz все releases:
   ```
   GET https://musicbrainz.org/ws/2/recording/{mbid}?inc=releases+release-groups&fmt=json
   ```
2. Анализируем `releases[]` и связанные `release-group.primary-type`:
   - Если найден release с `primary-type = "Album"` (или EP) → берём самый ранний по дате → это и есть альбом трека.
   - Если все releases — это `primary-type = "Single"` → трек действительно сингл.
   - Если releases отсутствуют или информация противоречивая → fallback (см. ниже).
3. Если AcoustID не дал mbid — пробуем поиск по `MusicBrainz search` (artist + title):
   - Берём top-1 recording с высоким score (≥ 90).
   - Повторяем шаг 2.
4. **Fallback (если уверенно определить альбом не удалось)**:
   - Не выдумываем альбом. Создаём `Album(is_single=true, title=<track title>)`.
   - Трек помечается флагом `metadata.album_unresolved=true` (jsonb в `tracks.metadata`), чтобы потом фоновая задача `reconcile_albums` могла повторить попытку.
5. **Найденный альбом**:
   - Ищем существующий `Album` по `mbid` (release-group). Если есть — переиспользуем.
   - Если нет — создаём `Album(is_single=false, title=<release.title>, mbid=<...>, release_year=<...>, cover_path=<из Cover Art Archive>)`.
   - Привязываем трек к этому альбому.

### Когда `is_single = true`
Только если MusicBrainz явно сообщил, что у recording'а **все** связанные release-group имеют `primary-type = "Single"` (или это единственный известный релиз с таким типом). В остальных случаях, включая «не удалось определить», флаг `is_single` остаётся `false` (т.е. трек просто принадлежит автогенерированному альбому-«контейнеру» до повторного reconcile).

### Конфликт с введёнными пользователем данными
Если пользователь всё же указал альбом вручную, а backend нашёл другой:
- При уровне доверия `high` (AcoustID exact) — показываем предложение в карточке верификации, но пользователь решает (кнопка «использовать MusicBrainz-данные»).
- При `medium/low` — приоритет у пользователя, версия MusicBrainz сохраняется как `metadata.album_suggestion`.

### Фоновая задача `reconcile_albums` (worker)
Запускается раз в сутки. Берёт треки с `metadata.album_unresolved=true` или с `is_single=true` без mbid и пробует повторить разрешение альбома (например, обновилась база MusicBrainz). При успехе — переносит трек в найденный альбом и снимает флаги.

Схема [`04-database-schema.md`](04-database-schema.md:1) использует `albums.is_single` строго по правилам выше.

## 3. Поиск трека в открытых источниках (source matching)

Цель — обогатить индекс реальной публичной ссылкой, чтобы при восстановлении на новом сервере (см. [`16-library-index-and-restore.md`](16-library-index-and-restore.md:1)) трек можно было перекачать без участия пользователя.

### Алгоритм

1. После загрузки файла backend имеет: `audio_sha256`, `acoustid_fingerprint`, предполагаемые `artist + title` (из тегов или пользовательской формы) и, возможно, `mbid` от AcoustID.
2. Формируется поисковый запрос: `"{artist} - {title}"` (плюс длительность как доп. фильтр).
3. Запускается параллельный поиск через тот же сервис, что и в [`06-music-sourcing.md`](06-music-sourcing.md:1):
   - `yt-dlp ytsearch10:<query>` → YouTube
   - `yt-dlp scsearch10:<query>` → SoundCloud
   - (опц.) Bandcamp по прямому поиску
4. Для каждого кандидата считается **score соответствия**:
   ```
   match = w1 * duration_match (1 − |Δduration|/Δduration_max, обнуляется при разнице >5 сек)
         + w2 * title_artist_similarity (token-set ratio)
         + w3 * isrc_match (если ISRC из тегов совпал с тем, что вернул источник — +0.3)
   ```
5. **Кандидат считается подтверждённым**, если `match ≥ 0.8` **И** разница длительности ≤ 2 сек. Берём top-3 таких кандидатов.
6. Если есть кандидаты — все они показываются пользователю; **primary** по умолчанию — тот, у кого выше score; пользователь может изменить выбор.
7. Если ни одного подтверждённого кандидата — индекс получает только `user_upload`. Это нормально; фоновая задача `match_user_uploads` повторит поиск позже (см. §6).

### Запись в `track_sources`

После подтверждения создаются записи:

| recovery_priority | source_kind | source_url | примечание |
|---|---|---|---|
| 10 | публичный (например, `youtube`) | URL и source_id | primary для восстановления |
| 20 | другие найденные публичные | URLs | резервные |
| 100 | `user_upload` | null | оригинальный файл пользователя; `original_filename`, `original_tags` |

При `library restore` сначала пробуются записи с меньшим `recovery_priority` (т.е. публичные источники), и только потом — пометка `needs_user_reupload` для `user_upload`. Поэтому, если найдены публичные ссылки, перенос на новый сервер не требует от пользователя ничего.

### Проверка целостности
После скачивания публичного кандидата backend **сравнивает** его `audio_sha256` (после транскода в opus 96kbps) с тем, что был у пользовательской загрузки. Совпадение не обязательно (битрейты разные), но **chromaprint fingerprint** должен совпадать с порогом ≥ 0.97. Если нет — кандидат отклоняется, в `track_sources` не пишется.

> Тяжёлая операция: скачивание делается **ленически** — только когда пользователь подтверждает публикацию (`confirm`). На этапе верификации показываются метаданные кандидатов без скачивания.

### Опциональное «promote»: использование публичного файла вместо пользовательского
В UI у master/admin есть кнопка «использовать публичный источник как основной файл»:
- Backend скачивает primary-источник, транскодит, заменяет физический файл (`storage/...`).
- Старый файл (от пользователя) удаляется (или архивируется).
- `tracks.audio_sha256` обновляется. `user_upload`-запись в `track_sources` остаётся как fallback.
- Зачем: единообразие битрейтов в библиотеке, экономия места.

## 4. Верификация достоверности

Цель — не пускать в общую библиотеку мусор и опечатки, при этом не мешать пользователю.

### Этапы верификации (по убыванию надёжности)

| Этап | Источник | Что даёт |
|---|---|---|
| 3.1 | **AcoustID + Chromaprint** | аудио-отпечаток → mbid → точное совпадение |
| 3.2 | **ShazamKit** (только iOS-клиент) | распознавание на устройстве, ISRC + title/artist |
| 3.3 | **MusicBrainz search** по artist+title | fuzzy совпадение, проверка существования |
| 3.4 | **Last.fm** | подтверждение артиста и жанров |
| 3.5 | **Heuristics** | длительность, частота, битрейт в разумных пределах |

### Расчёт уровня доверия

```
trust = max(
  acoustid_match,            # 0..1, exact = 1.0
  shazam_match,              # 0..1
  musicbrainz_fuzzy,         # 0..1, по similarity названия+артиста
)
+ small_bonus * (наличие_тегов_в_файле, валидный_год, валидная_длина)
```

| Уровень | Условие | Действие |
|---|---|---|
| `high` | trust ≥ 0.85 (есть mbid из AcoustID или MB exact) | предложить «принять как есть» одной кнопкой |
| `medium` | 0.5 ≤ trust < 0.85 | показать предложения от MB, пользователь подтверждает |
| `low` | 0.2 ≤ trust < 0.5 | предупредить «возможны ошибки», требовать подтверждение чекбоксом |
| `none` | trust < 0.2 | не блокировать (некоммерческий сервис), но пометить трек тегом `unverified` и показывать иконку в каталоге |

### Реализация
- Сервис `app/services/verification/` с шагами 4.1–4.5.
- Параллельно — сервис `app/services/source_matching/` (см. §3).
- Запуск асинхронно (oба процесса в одной upload-сессии).
- Кеширование результатов: `(audio_sha256) → verification_payload` и `match_payload` в Redis на 24ч.
- На macOS-сервере: бинарь `fpcalc` (chromaprint) в Docker-образе backend.

### API

```
POST /api/v1/tracks/upload
  multipart: file=<audio>, cover=<image?>, meta=<json>
  → { upload_id, parsed_meta, verification_pending: true }

GET /api/v1/tracks/upload/{upload_id}/verify
  → {
      trust_level: "high|medium|low|none",
      score: 0.92,
      suggestions: { title?, artist?, album?, year?, mbid?, genres?[] },
      sources_verification: ["acoustid", "musicbrainz"],
      candidate_sources: [
        {
          kind: "youtube", source_id: "kXYi...", url: "...",
          match: 0.94, duration_delta: 1,
          title: "...", channel: "...", thumbnail: "..."
        },
        ...
      ],
      primary_source_index: 0
    }

POST /api/v1/tracks/upload/{upload_id}/confirm
  body: {
    meta: {...},
    accept_suggestions: bool,
    force: bool,
    selected_source_ids: ["youtube:kXYi...", ...],   // что записать в track_sources
    primary_source_id: "youtube:kXYi..."             // основной для restore
  }
  → { track_id }
```

Если пользователь не подтвердил в течение 30 минут — временный файл и upload-сессия удаляются (worker `cleanup_uploads`).

### Анти-дубль
Перед созданием Track: сверка по `audio_sha256` и (если найден) `mbid`. Дубликат не создаём, возвращаем существующий с пометкой `already_exists=true`. Если найден дубликат, а у него ещё нет публичных `track_sources` — добавляем найденные кандидаты к существующему треку (обогащаем индекс).

- Та же форма.

## 5.1 Фоновая задача `match_user_uploads`
Раз в сутки (или по запросу master) проходит по трекам, у которых в `track_sources` есть только `user_upload`, и повторяет источник-матчинг через §3. Полезно когда:
- На момент загрузки трек ещё не был доступен в публичных источниках (например, только что вышел сингл — попал позже).
- В первый раз поиск ничего не нашёл, но позже YouTube/SoundCloud получил подходящее видео.

Это превентивно «дозаливает» индекс, чтобы при будущем `library restore` максимум треков восстанавливалось автоматически.
